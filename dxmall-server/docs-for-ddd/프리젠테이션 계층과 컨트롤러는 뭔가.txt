프리젠테이션 계층(Controller)은 왜 서비스와 분리하는가?

핵심: 같은 비즈니스 로직을 여러 방식으로 호출할 수 있게 하기 위해서다.

예를 들어 "회원가입"이라는 기능은 하나인데, 호출하는 방식은 여러 가지다:
- REST API (POST /api/auth/signup)
- GraphQL
- 메시지 큐 (Kafka 이벤트로 가입 처리)
- CLI (관리자가 터미널에서 직접 가입 처리)

만약 컨트롤러에 비즈니스 로직을 다 넣으면:
- REST API용 회원가입 로직
- GraphQL용 회원가입 로직
- Kafka용 회원가입 로직
- 전부 따로 작성해야 하고, 규칙이 바뀌면 전부 고쳐야 한다.

분리하면:
  Controller (프리젠테이션)  → "HTTP 요청을 받아서 서비스에 전달"
  AuthService (응용)         → "중복확인 → 유저생성 → 토큰발급" (한 번만 작성)

  REST Controller  → AuthService.signup() 호출
  GraphQL Resolver → AuthService.signup() 호출
  Kafka Listener   → AuthService.signup() 호출
  전부 같은 서비스를 재사용한다.

컨트롤러의 역할:
1. 외부 요청을 받는다 (HTTP, GraphQL 등)
2. 요청 데이터를 서비스가 이해할 수 있는 형태(DTO)로 변환한다
3. 서비스를 호출한다
4. 결과를 외부 응답 형태(JSON 등)로 변환해서 돌려준다

즉, 컨트롤러는 "요청/응답 변환기"이고,
비즈니스 로직은 서비스에만 있어야 어디서든 재사용할 수 있다.

"프리젠테이션"이라는 이름이 붙은 이유:
외부 세계와 만나는 접점이라는 뜻이지만,
실질적으로는 "호출 방식(프로토콜)을 담당하는 계층"이라고 이해하면 된다.

보충 설명:
이를테면 컨트롤러 서비스 계층을 분리하는 이유는 요청을 받아서 응답하기까지의 일련의 로직에서
상세한 실행 흐름은 서비스 계층으로 분리하겠다는 의미다.
